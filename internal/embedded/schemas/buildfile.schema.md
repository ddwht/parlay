# Buildfile Schema

File: `.parlay/build/<feature-name>/buildfile.yaml`
Generated by `/parlay build-feature`. Tool-internal prototype specification — never user-facing.

The buildfile is the deterministic bridge between design artifacts and generated code. Same buildfile = same functionality, same layout, same patterns — regardless of which AI agent generates the code.

The buildfile structure is **framework-agnostic**. Framework-specific vocabulary (widget names, layout strategies, interaction styles) comes from the **framework adapter** loaded at generation time.

## Structure

```yaml
feature: <feature-slug>
adapter: <framework-adapter-name from .parlay/adapters/>

models:
  <EntityName>:
    properties:
      <property-name>:
        type: <string | number | boolean | date | enum | EntityName | EntityName[]>
        enum: [<values>]
        required: <boolean>
        description: <what this property represents>
    states:
      values: [<state1>, <state2>, ...]
      transitions:
        <from-state>: [<to-state1>, <to-state2>]

fixtures:
  <fixture-name>:
    description: <scenario this fixture represents>
    data:
      <EntityName>:
        - <object matching model properties>

routes:
  - path: <route identifier>
    page: <page name matching surface Page targets>
    layout-pattern: <from adapter — e.g., "page-with-sidebar", "sequential-prompts">
    regions:
      <region-name>:
        components: [<component references in order>]

components:
  <component-name>:
    source: <@feature/fragment-name from surface>
    type: <abstract component type — see below>
    widget: <framework-specific widget from adapter>
    description: <what this component does>

    data:
      inputs:
        - name: <property name>
          model: <EntityName>
          field: <property path>
          from: <data source — command-argument, file-read, computed, fixture>
      computed:
        - name: <derived value name>
          from: <derivation description>
          type: <result type>

    elements:
      - name: <element identifier>
        type: <abstract element type — see below>
        widget: <framework-specific widget from adapter>
        bind: <data property or computed value>
        format: <display format>
        visible-when: <condition or "always">

    actions:
      - name: <action identifier>
        type: <abstract action type — see below>
        widget: <framework-specific widget from adapter>
        enabled-when: <condition or "always">
        effect: <what happens — navigate, mutate, emit, call>
        target: <route, model.field, event-name, or operation-name>

    operations:
      - type: <operation type — see Operation types below>
        <type-specific parameters>

    children:
      - <nested component-name references>
```

## Abstract types

These are framework-agnostic. The adapter maps each to a framework-specific widget.

### Component types

| Type | Description |
|---|---|
| interactive-wizard | Multi-step guided flow with sequential inputs |
| command-output | Executes an operation and displays results |
| interactive-prompt | Asks the user a question and processes the answer |
| data-display | Shows data without user interaction |
| report | Structured output with multiple sections |

### Element types

| Type | Description |
|---|---|
| text-output | Displays a text message |
| data-list | Displays a list of items |
| data-table | Displays tabular data |
| status-indicator | Shows a state value (success, warning, error) |
| path-reference | Displays a file or directory path |
| grouped-output | Groups related items under headings |
| code-block | Displays formatted code or structured content |

### Action types

| Type | Description |
|---|---|
| selection | Choose from a list of options |
| confirmation | Yes/no or approve/reject |
| text-input | Free-form text entry |
| navigation | Move to a different route/view |
| file-operation | Create, read, write, or delete a file |
| state-mutation | Change model state |

### Effect syntax

| Effect | Syntax | Description |
|---|---|---|
| Navigate | `navigate:<route>` | Move to a route |
| Mutate | `mutate:<Model>.<field>=<value>` | Change model state |
| Emit | `emit:<event-name>` | Signal to other components |
| Call | `call:<operation-name>` | Invoke a domain operation |

## Operation types

The `operations:` section uses a closed set of typed operations. Free-form pseudo-code is not allowed — every operation must declare a `type:` and the parameters required by that type. This guarantees the determinism contract: two agents reading the same buildfile produce structurally identical code.

| Type | Required parameters | Optional parameters | Description |
|---|---|---|---|
| `read-file` | `path` | `parse`, `bind` | Read a file from disk. `parse` selects a schema (intent-schema, surface-schema, etc.) for structured parsing. `bind` names the variable holding the result. |
| `write-file` | `path`, `content` | `template`, `for-each` | Write content to a file. `template` references a content template. `for-each` iterates over a bound variable. |
| `create-directory` | `path` | — | Create a directory (recursive, like mkdir -p). |
| `copy` | `from`, `to` | `recursive` | Copy a file or directory. `recursive` enables directory copy. |
| `filter` | `source`, `where`, `bind` | — | Filter a bound collection by a predicate, store result in a new variable. |
| `for-each` | `source`, `do` | — | Iterate over a collection. `do` is a list of nested operations. |
| `transform` | `source`, `using`, `bind` | — | Transform data using a named transformer (e.g., slugify, json-encode), store result in a variable. |
| `call-skill` | `skill` | `args` | Invoke another skill by name with optional arguments. |

### Examples

```yaml
operations:
  - type: read-file
    path: spec/intents/{feature}/intents.md
    parse: intent-schema
    bind: intents

  - type: filter
    source: intents
    where: "title not in existing-content"
    bind: new-intents

  - type: for-each
    source: new-intents
    do:
      - type: write-file
        path: spec/intents/{feature}/dialogs.md
        template: dialog-template
        for-each: new-intents

  - type: create-directory
    path: .parlay/

  - type: copy
    from: embedded:schemas
    to: .parlay/schemas/
    recursive: true
```

### Variable bindings

- Variables created by `bind:` are scoped to the component's `operations:` block.
- Path templates support `{feature}`, `{slug}`, and any bound variable in `{var.field}` form.
- Predicates in `filter.where` use simple boolean expressions: `field == value`, `field in collection`, `field not in collection`.

## Determinism contract

Two AI agents reading the same buildfile must produce prototypes where:
- The same components exist with the same nesting
- The same data is displayed in the same elements
- The same actions are available with the same triggers
- The same navigation structure works
- The same fixtures produce the same visible output
- The same framework-specific widgets are used (as specified in the buildfile)

## Parsing

- YAML structure
- Model references: `<EntityName>` in data.inputs.model and fixtures
- Component references: `<component-name>` in routes and children
- Fragment references: `@feature/fragment-name` in source
- Condition expressions: string values in `visible-when` and `enabled-when`
- Effect expressions: `type:target` pattern in action effects
- Adapter reference: `adapter` field links to `.parlay/adapters/<name>.adapter.yaml`
