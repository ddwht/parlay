# Design Spec Schema

File: `.parlay/build/<feature-name>/design-spec.yaml`
Generated by `/parlay-reference-design-spec`. Tool-internal — same visibility as buildfile.yaml.

The design-spec captures **per-fragment visual design details** extracted from a Figma file. It enriches what the surface deliberately omits: exact widget variants, layout dimensions, token references, state-specific visuals, spacing, and colors.

The design-spec is **optional**. When it exists, build-feature uses it to produce a richer buildfile. When it doesn't exist, adapter defaults apply unchanged. The pipeline works identically either way.

## Structure

```yaml
feature: <feature-slug>
figma-source: <Figma file URL or identifier>
generated: <ISO date>

shared:
  tokens:
    <category>: <token references applying to all fragments>
  spacing:
    - <shared spacing values>
  colors:
    - <shared color references>

fragments:
  <fragment-name>:
    figma-source: <Figma frame/node identifier>
    widget: <exact framework widget variant>
    layout:
      - <columns, widths, flex ratios, alignment, row heights>
    tokens:
      <category>: <specific token references>
    variants:
      <state>: <visual treatment>
    spacing:
      - <padding, gaps, margins>
    colors:
      - <specific color references>
```

## Fields

### Global fields

| Field | Required | Description |
|---|---|---|
| `feature` | Yes | Feature slug — must match the feature directory name |
| `figma-source` | Yes | Figma file URL or identifier for traceability |
| `generated` | No | ISO date of generation |

### Shared section

Optional. Values here apply to all fragments unless overridden per-fragment.

| Field | Required | Description |
|---|---|---|
| `tokens` | No | Map of category name → token references. Categories should match the adapter's `design-system:` categories. |
| `spacing` | No | List of spacing values shared across fragments |
| `colors` | No | List of color token references shared across fragments |

### Per-fragment fields

Each key under `fragments:` must match a fragment name from the feature's `surface.md`. Fragments in the surface that have no Figma counterpart are simply absent — they fall through to adapter defaults.

| Field | Required | Description |
|---|---|---|
| `figma-source` | No | Figma frame or node identifier for this specific fragment |
| `widget` | No | Exact framework widget variant — more specific than the adapter's generic mapping. E.g., "Table with fixed header and striped rows" vs just "Table". |
| `layout` | No | List of layout details: column definitions, widths, flex ratios, alignment, row heights, grid structure |
| `tokens` | No | Map of category name → token references specific to this fragment. Overrides shared tokens for the same category. |
| `variants` | No | Map of state name → visual treatment. Common states: `loading`, `error`, `empty`, `hover`, `focus`, `disabled`. |
| `spacing` | No | List of spacing values: padding, gaps, margins between elements within this fragment. Overrides shared spacing. |
| `colors` | No | List of color token references for status indicators, backgrounds, accents, borders. Overrides shared colors. |

## Token category references

Token category names in the `tokens:` field should reference categories declared in the adapter's `design-system:` section. This is the link between the design-spec and the adapter:

- Adapter declares `colors: { source: figma }` → design-spec provides the actual Figma color values per fragment. These are authoritative — build-feature uses them directly.
- Adapter declares `spacing: { source: framework }` → design-spec values MUST be framework token references, not raw Figma values. The reference-design-spec skill maps Figma visual properties to the closest framework token during extraction. Build-feature uses these mapped token references. If a value cannot be mapped to a framework token, it is omitted from the design-spec (the framework default applies).
- Adapter declares `motion: { source: not-defined }` → design-spec can provide motion values that the framework doesn't cover. Values are recorded as-is from Figma.

**Contract**: For `source: framework` categories, the design-spec never contains raw hex values, Figma token names, or custom class names. It only contains framework token references that match the adapter's `format` field. This invariant is maintained by the reference-design-spec skill during extraction — the user confirms the mapping before the design-spec is written.

When a design-system category has `source: figma` but the design-spec has no corresponding value for a fragment, the agent treats it as `not-defined` for that fragment and uses sensible defaults.

## Variant states

The `variants:` field maps state names to visual treatments. These go beyond what surface's `visible-when` covers — they describe HOW something looks in each state, not just whether it's visible.

Common states:

| State | What it describes |
|---|---|
| `loading` | Visual treatment while data is being fetched (skeleton, spinner, shimmer) |
| `error` | Visual treatment when an operation fails (error illustration, red border, retry button) |
| `empty` | Visual treatment when no data exists (illustration, message, CTA) |
| `hover` | Visual treatment on mouse hover (highlight, elevation, cursor change) |
| `focus` | Visual treatment on keyboard focus (ring, outline, border change) |
| `disabled` | Visual treatment when interaction is unavailable (opacity, grey-out) |

Teams can add custom states beyond this set.

## Shared vs per-fragment precedence

Per-fragment values override shared values for the same field:
- If `shared.tokens.colors` is set AND `fragments.task-board.tokens.colors` is set, the per-fragment value wins for the task-board fragment
- Other fragments without a per-fragment override use the shared value
- If neither shared nor per-fragment provides a value, the adapter default applies

## Validation

When a design-spec file is loaded, the tool verifies:
- Valid YAML syntax
- `feature` field matches the feature directory name
- Every key under `fragments:` matches a fragment name in the corresponding `surface.md`
- Fragment keys not present in surface.md are errors (stale reference)
- Token category names match categories declared in the adapter's `design-system:` section (warning, not error — adapter may be updated later)

## Relationship to other artifacts

| Artifact | Relationship |
|---|---|
| **Surface** | Upstream. Surface fragment names are the join keys. Design-spec enriches but never contradicts the surface's interaction model. |
| **Adapter (design-system)** | Sibling. Adapter declares which categories exist and their source. Design-spec provides values for categories with `source: figma`. |
| **Buildfile** | Downstream. Build-feature reads design-spec and produces a richer buildfile. Design-spec values flow INTO the buildfile, not around it. |
| **Figma** | Source. Design-spec is extracted from Figma via MCP. The `figma-source` fields maintain traceability. |

## Pipeline consumption

**reference-design-spec** generates the design-spec by reading Figma via MCP and mapping components to surface fragments.

**build-feature** reads the design-spec (if it exists) alongside intents, dialogs, surface, domain-model, adapter, and blueprint. It uses the design-spec to:
- Refine component `widget:` values in the buildfile
- Add layout properties to elements
- Add token references (especially for `source: figma` categories)
- Add state-specific elements from variants
- Add spacing and color values

**generate-code** never reads design-spec directly — the richer details are already in the buildfile. The codegen boundary is preserved.
