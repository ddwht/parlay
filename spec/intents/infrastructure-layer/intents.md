# Infrastructure Layer

> Extend the parlay feature structure with a framework-agnostic `infrastructure.md` artifact, so that features with behind-the-scenes changes can be described in the same portable, framework-independent way as surfaces — and translated to framework-specific code at build time via the adapter. Infrastructure.md describes WHAT behavioral capabilities a feature needs; the buildfile's `cross-cutting:` section (generated by build-feature) describes HOW to implement them in the current framework. This mirrors the surface pipeline: surface.md (Shows/Actions) → adapter → buildfile components (widgets). See also: `qualified-identifier-resolver` for the `cross-cutting:` buildfile section schema.

---

## Define the Infrastructure Artifact

**Goal**: Introduce `infrastructure.md` as a framework-agnostic artifact in the parlay feature structure, parallel to `surface.md`, so that behind-the-scenes changes can flow through the full pipeline without requiring a user-facing surface — and without binding the design to a specific framework, language, or codebase layout.
**Persona**: Parlay Developer
**Priority**: P0
**Context**: The parlay pipeline assumes every feature has a `surface.md` that describes what the user sees. Build-feature maps surface fragments to buildfile components. But some features have no user-facing surface — they add classifiers, shared helpers, or validation logic. Today these features hit a dead end: check-readiness blocks on "no surface," and build-feature has nothing to map. A parallel artifact for infrastructure changes closes this gap. Critically, `surface.md` is framework-agnostic — it uses vocabulary (Shows, Actions, Flows) that the adapter translates to framework-specific widgets. `infrastructure.md` must follow the same principle: it describes behavioral requirements that the adapter translates to framework-specific code patterns. This ensures the same intents, dialogs, surface, and infrastructure can be used to generate code for any framework — a Go CLI, a React admin panel, a Python service — by swapping the adapter.
**Action**: Add `spec/intents/<feature>/infrastructure.md` as an optional file in the feature structure. It describes behavioral capabilities — what the code must do — using framework-agnostic language. It does NOT name specific functions, file paths, type signatures, or language constructs. Those details are resolved at build-feature time, when the adapter and existing source tree are available.
**Objects**: infrastructure, feature-structure, pipeline, portability

**Constraints**:
- `infrastructure.md` is optional. Features that are purely user-facing use only `surface.md`. Features that are purely behind-the-scenes use only `infrastructure.md`. Features with both have both files.
- The file belongs to the same ownership zone as `surface.md`: generated by the tool (or the agent), reviewed by the designer. It lives at `spec/intents/<feature>/infrastructure.md`.
- Each infrastructure fragment describes one discrete behavioral capability — a classifier, a validation guard, a traversal change, a caching strategy. Fragments are separated by `---`, just like surface fragments.
- Infrastructure fragments trace back to their source intent via a `**Source**:` field, just like surface fragments. Every fragment must have a source.
- Infrastructure fragments are **framework-agnostic**. They must NOT contain: function names or signatures (e.g., `classifyDir(path string)`), file paths (e.g., `internal/config/config.go`), language-specific type declarations (e.g., `type DirClass int`), grep patterns targeting framework identifiers (e.g., `hasIntentsMd|AllFeaturePaths`), or import paths. These are implementation details that belong in the buildfile's `cross-cutting:` section, generated at build-feature time by consulting the adapter and scanning the source tree.
- Infrastructure fragments MAY reference domain concepts from the intents (e.g., "feature", "initiative", "intents.md", "qualified identifier") because these are part of the problem domain, not the framework. The line is: domain vocabulary is allowed; implementation vocabulary is not.
- Check-readiness at the build-feature stage must accept a feature that has `infrastructure.md` but no `surface.md`. The rule becomes: at least one of the two must exist, not both required.
- The pipeline path for a pure infrastructure feature is: intents → infrastructure.md (authored directly or generated from intents by the agent) → build-feature (translates to cross-cutting via adapter) → generate-code.
- Infrastructure fragments do not use the surface vocabulary (Shows, Actions, Flows). They have their own field vocabulary defined in `infrastructure.schema.md`.
- The portability guarantee: the same `infrastructure.md` file, combined with a different adapter, should produce valid `cross-cutting:` entries targeting that adapter's framework. A Go CLI adapter resolves "directory classifier" to a Go function in config.go. A Python adapter resolves it to a Python class in a resolver module. The infrastructure.md itself is unchanged.

**Verify**:
- A feature with only `infrastructure.md` (no `surface.md`) passes check-readiness at the build-feature stage
- A feature with only `surface.md` (no `infrastructure.md`) continues to work exactly as before
- A feature with both `surface.md` and `infrastructure.md` is accepted by check-readiness
- A feature with neither `surface.md` nor `infrastructure.md` fails check-readiness with an error asking the designer to create at least one
- An `infrastructure.md` file that names a specific function, file path, or type signature fails validation with a warning about framework-specific content
- The same `infrastructure.md` combined with two different adapters (e.g., go-cli and a hypothetical python-cli) produces different `cross-cutting:` entries targeting each framework's conventions — the infrastructure file itself is identical

## Define the Infrastructure Schema

**Goal**: Create `infrastructure.schema.md` that defines the fragment format, field vocabulary, and validation rules for the framework-agnostic `infrastructure.md` — giving it the same rigor as `surface.schema.md` gives to surfaces, while ensuring no framework-specific content leaks into the artifact.
**Persona**: Parlay Developer
**Priority**: P0
**Context**: Without a schema, infrastructure fragments are free-form text that the agent interprets on every run — non-deterministic and error-prone. The schema pins the field names, their meanings, which are required, and how they feed build-feature's translation to `cross-cutting:` entries. The schema also enables `parlay validate --type infrastructure` for structural validation. The key design decision: infrastructure.md fields describe WHAT (behavioral requirements) — the buildfile's cross-cutting fields describe HOW (framework-specific implementation). The schema enforces this boundary.
**Action**: Create `.parlay/schemas/infrastructure.schema.md` (and its source at `internal/embedded/schemas/infrastructure.schema.md`) with the fragment format, field definitions, and validation rules. Deploy via `make sync-skills`.
**Objects**: schema, infrastructure, validation, portability

**Constraints**:
- The schema defines these fields per fragment:

  | Field | Required | Description |
  |---|---|---|
  | Fragment Name | Yes | `## ` heading, unique within feature. Names the capability (e.g., "Directory Classification", "Slug Uniqueness Guard"). |
  | Affects | Yes | Abstract scope of the change — what area of the system this capability touches. Domain-level labels, not file paths or function names. Examples: "feature resolution", "directory traversal", "slug validation", "command dispatch". Free-form text; multiple scopes comma-separated. |
  | Behavior | Yes | Human-readable description of what the capability does — inputs, outputs, rules, error conditions. Must be framework-agnostic: describe the WHAT, never the HOW. |
  | Invariants | No | `- ` prefixed testable properties that must hold after the capability is implemented. These feed the testcases for cross-cutting entries. Each invariant is a falsifiable statement about system behavior. |
  | Source | Yes | `@feature/intent-slug` reference. Every fragment must trace to its source intent. |
  | Caching | No | Abstract caching strategy: `on-first-access`, `none`, `per-process`, or a custom description. Describes the caching behavior, not the implementation mechanism (`sync.Once` is Go-specific and not allowed here). |
  | Backward-Compatible | No | `yes` or `no`. Whether existing callers of affected code must continue working without changes. |
  | Notes | No | `- ` prefixed lines for additional constraints, design decisions, edge cases. |

- The following fields from the previous schema are **removed** because they are framework-specific and now belong to the buildfile's `cross-cutting:` section (generated at build-feature time):
  - `Modifies` — replaced by `Affects` (abstract scope instead of concrete file/function references)
  - `Introduces` — removed entirely (new function signatures are framework-specific; the Behavior field describes what capability is introduced)
  - `Detection` — removed entirely (grep patterns are framework-specific; build-feature resolves targets by scanning the source tree with adapter conventions)

- `Affects` is required because every infrastructure fragment must scope what area of the system it touches. Without it, build-feature has no starting point for finding target files. `Affects` is deliberately more abstract than `Modifies`: "feature resolution" rather than `config.FeaturePath`; "directory traversal" rather than `AllFeaturePaths`; "slug validation" rather than `resolveQualifiedPath`. The agent resolves these to concrete files at build time using the adapter's `file-conventions` and source tree scanning.
- `Behavior` remains the primary payload. It must describe the capability in framework-agnostic terms. Validation should warn (not error) on content that looks framework-specific — function signatures with type annotations, file paths with extensions, language keywords like `func`, `def`, `class`, `interface`.
- `Invariants` are optional but recommended. Each invariant is a testable property: "A directory with intents.md at its root is always classified as Feature", "Two directories slugifying to the same identifier under the same parent produce an error listing both paths." Build-feature uses invariants to generate testcases for cross-cutting entries.
- The schema follows the same structure as `surface.schema.md`: template, fields table, validation rules, parsing rules.
- Validation (`parlay validate --type infrastructure`) checks: fragment names are unique, every fragment has Affects, every fragment has Behavior, every fragment has Source, Source references point to existing intents, Backward-Compatible value is `yes` or `no` when present. Additionally, a portability lint warns on content that appears framework-specific (file extensions, function signatures, language-specific type syntax).
- The schema is deployed to `.parlay/schemas/infrastructure.schema.md` via `make sync-skills`.

**Verify**:
- `parlay validate --type infrastructure --json spec/intents/<feature>/infrastructure.md` passes for a well-formed file and returns structured errors for malformed fragments
- A fragment missing `Behavior` fails validation
- A fragment missing `Affects` fails validation
- A fragment missing `Source` fails validation
- A fragment with Behavior containing `func classifyDir(path string)` triggers a portability warning
- A fragment with Affects containing `internal/config/config.go` triggers a portability warning
- A well-formed fragment with only framework-agnostic content passes validation cleanly
- The schema file exists at both `internal/embedded/schemas/infrastructure.schema.md` (source) and `.parlay/schemas/infrastructure.schema.md` (deployed) after `make sync-skills`

## Bridge Infrastructure to Framework-Specific Cross-Cutting

**Goal**: Extend `/parlay-build-feature` to translate framework-agnostic infrastructure fragments into framework-specific `cross-cutting:` entries in the buildfile — the same way it translates surface Shows/Actions into framework-specific widgets via the adapter. This is the adapter bridge for infrastructure.
**Persona**: Parlay Developer
**Priority**: P0
**Context**: Surface fragments say `data-list` and the adapter says "that's a `bulleted-list` in Go CLI." Infrastructure fragments say "directory classification capability that distinguishes features from initiatives" and build-feature must now say "that means modifying `AllFeaturePaths` in `internal/config/config.go`, introducing a `classifyDir` function, and using a pattern scan for `hasIntentsMd`" — consulting the adapter's file conventions and the existing source tree to make these decisions. The translation is less mechanical than surface-to-widget mapping (infrastructure is more varied), so the agent's judgment plays a larger role, guided by the adapter's conventions and the fragment's Affects/Behavior fields.
**Action**: Update the build-feature skill to: (1) read `infrastructure.md` if it exists, (2) for each infrastructure fragment, consult the adapter and scan the source tree to resolve abstract Affects/Behavior into concrete cross-cutting entry fields (target-files, target-pattern, introduces, transform), (3) generate `cross-cutting:` entries in the buildfile. Also update check-readiness to accept features with only `infrastructure.md`.
**Objects**: build-feature, infrastructure, buildfile, cross-cutting, adapter, resolution

**Constraints**:
- The translation from infrastructure fragment to buildfile `cross-cutting:` entry is NOT a direct field mapping — it is an agent-mediated resolution that uses the adapter and source tree as context. The agent:
  1. Reads the fragment's `Affects` field to determine what area of the codebase to scan (e.g., "feature resolution" → scan for path-resolution functions under the adapter's `file-conventions.source-root`)
  2. Reads the fragment's `Behavior` field to understand what the change does
  3. Consults the adapter's `conventions` and `file-conventions` to determine naming patterns, file organization, and code style for the target framework
  4. Scans the existing source tree to find concrete files that match the abstract Affects scope
  5. Generates a `cross-cutting:` entry with framework-specific fields:
     - `target-files:` — resolved from Affects + source tree scan
     - `target-pattern:` — generated when the Behavior describes a fan-out change, using framework-specific identifiers found in the source tree
     - `introduces:` — generated from the Behavior description, using framework-specific naming conventions from the adapter
     - `transform:` — synthesized from Behavior + Invariants + Notes, enriched with framework-specific context
     - `source:` — passed through from the infrastructure fragment's Source
  6. Uses `Invariants` from the fragment to generate testcases for the cross-cutting entry
- The resolution is adapter-aware: different adapters produce different cross-cutting entries from the same infrastructure fragment. A Go CLI adapter resolves "directory classification" to a Go function with error returns. A Python adapter resolves it to a Python class with exception handling. The infrastructure fragment is identical in both cases.
- Build-feature reads `infrastructure.md` AFTER reading `surface.md` (if present). Both contribute to the same buildfile. Infrastructure fragments produce only `cross-cutting:` entries, never `components:` or `routes:`.
- If a feature has `infrastructure.md` but no `surface.md`, the buildfile has `cross-cutting:` entries but no `components:` or `routes:`. This is valid.
- If a feature has both, the buildfile has `components:` (from surface) AND `cross-cutting:` (from infrastructure).
- The `cross-cutting:` entries follow the same diff lifecycle as components: `parlay diff` classifies each as stable/dirty/removed based on whether the source infrastructure fragment changed.
- Deep validation (`parlay validate --type buildfile --deep`) validates the generated `cross-cutting:` entries: each `source:` traces to a real intent, each `target-files:` path exists (or `target-pattern:` matches at least one file).
- When the agent cannot confidently resolve an Affects scope to concrete files (no matching files found, ambiguous matches), it must ask the designer via AskUserQuestion rather than guessing. This is analogous to surface generation asking about ambiguous Show/Action mappings.

**Verify**:
- Running `/parlay-build-feature` on a feature with only `infrastructure.md` produces a buildfile with `cross-cutting:` entries and no `components:`
- Running `/parlay-build-feature` on a feature with both `surface.md` and `infrastructure.md` produces a buildfile with both `components:` and `cross-cutting:` entries
- The generated `cross-cutting:` entries contain framework-specific content (function names, file paths, detection patterns) derived from the adapter and source tree — even though the infrastructure.md contained none
- Running `/parlay-build-feature` with the same `infrastructure.md` but a different adapter produces different `cross-cutting:` entries targeting the new framework's conventions
- `parlay validate --type buildfile --deep` passes for the generated buildfile
- `parlay diff` includes `cross-cutting:` entries in its stable/dirty/removed output
- Check-readiness passes for a feature with `infrastructure.md` but no `surface.md`
- When Affects cannot be resolved to any file in the source tree, build-feature asks the designer which files are affected rather than silently generating a broken entry

## Update Generate-Code to Process Infrastructure-Sourced Entries

**Goal**: Ensure `/parlay-generate-code` processes `cross-cutting:` entries sourced from `infrastructure.md` the same way it processes entries from any other source — applying Tier 2 intelligent merge against target files, presenting diffs for review, and tracking state through the diff lifecycle.
**Persona**: Parlay Developer
**Priority**: P0
**Context**: The `qualified-identifier-resolver` feature defines how generate-code processes `cross-cutting:` entries in general (step ordering, Tier 2 merge, diff presentation). This intent ensures that entries produced from `infrastructure.md` fragments — now with framework-specific content generated at build time rather than authored manually — flow through the same mechanism without special-casing, and that the end-to-end pipeline works for pure infrastructure features (no components, only cross-cutting).
**Action**: Verify and document that generate-code's cross-cutting processing step handles infrastructure-sourced entries. The entries are now richer (build-feature resolved abstract Affects to concrete files, generated introduces from Behavior), so generate-code has everything it needs without reading back to `infrastructure.md`. The buildfile is the single source of truth at this stage.
**Objects**: generate-code, cross-cutting, intelligent-merge, buildfile

**Constraints**:
- Infrastructure-sourced `cross-cutting:` entries are indistinguishable from any other `cross-cutting:` entry at the buildfile level. Generate-code does not need to know the entry's origin — it processes all entries uniformly. This is the same isolation rule that applies to components: generate-code reads only the buildfile, never `spec/intents/`.
- For `target-pattern:` entries, generate-code greps the source tree to resolve a concrete file list. If zero files match, warn but don't error. If one or more match, apply Tier 2 intelligent merge to each.
- For `target-files:` entries, generate-code reads each named file directly. If a file doesn't exist and the entry has `introduces:`, create a new file with a `parlay-section: cross-cutting` marker. If a file doesn't exist and the entry only modifies, error.
- Cross-cutting entries use `parlay-section: cross-cutting` markers on new files and `parlay-extends:` markers when merging into existing component files.
- Cross-cutting entries follow the same diff lifecycle as components. On subsequent runs, `parlay diff` classifies each as stable/dirty/removed.
- `parlay save-build-state` captures cross-cutting state alongside component state.
- A pure infrastructure feature (only `cross-cutting:` entries, no components) must work end-to-end: build-feature → generate-code → tests → save-build-state.
- Process cross-cutting entries AFTER component generation and brownfield mount, but BEFORE tests — so infrastructure is in place when tests exercise the components that depend on it.

**Verify**:
- A buildfile with two `cross-cutting:` entries (one targeting a specific file, one with a `target-pattern:` matching three files) produces four Tier 2 intelligent merge diffs when generate-code runs
- Each diff is presented with the A/B/C review menu before being applied
- A `target-pattern:` matching zero files produces a warning, not an error
- A `target-files:` entry where the file doesn't exist AND the entry only modifies produces an error
- A `target-files:` entry where the file doesn't exist AND the entry has `introduces:` creates a new file
- Cross-cutting changes are tracked by `parlay diff` as stable/dirty/removed on subsequent runs
- After a successful end-to-end run, `parlay save-build-state` captures cross-cutting state
- A pure infrastructure feature completes the full pipeline: build-feature → generate-code → tests → save-build-state
- Generate-code never reads `infrastructure.md` — it reads only the buildfile. The framework-agnostic-to-specific translation happened at build-feature time.
